什么是JUC

JUC (为java.util.concurrent的简称) 是java并发编程中的工具类 为了更好的支持高并发任务 也是面试高频问题

JUC作为JDK5.0新特新添加到JDK中

包含如下三个包:

  • java.util.concurrent
  • java.util.concurrent.atomic
  • java.util.concurrent.locks

知识回顾

进程和线程

  • 进程
    • 进程是一个具有一定独立功能的程序关于某个数据集合的一次运行活动。
    • 是操作系统动态执行的基本单元，在传统的操作系统中，进程既是基本的分配单元，也是基本的执行单元
  • 线程
    • 通常在一个进程中可以包含若干个线程 (至少一个 否则没有存在的意义)
    • 线程可以利用进程所有拥有的资源
    • 在引入线程的操作系统中 通常都是把进程作为分配资源的基本单位 是独立运行和调度的基本单位
    • 系统对线程的调度付出开销小很多

在windows下可以查看任务管理器

例如: 用Word写文档的时候

  • Word进程为用户提供服务
  • Word后有某线程能帮用户自动保存

并发和并行

  • 并发
    • 同一个时间片只有一个程序正在运行
    • 不同任务在一个时间间隔(一小段时间)看起来像在同时运行 实际是多个任务串行运行 系统在调度
    • 逻辑意义上的同时运行(物理意义是串行)
    • 单CPU系统可以并发(模拟多个CPU) 相当于一个人在一个时间间隔内做很多件事情
  • 并行
    • 同一个时间片有多个程序同时运行
    • 逻辑意义和物理意义上都是同时运行
    • 只能多CPU系统下 相当于有多个人同时做多件事情 各自都在做事情

wait()方法和sleep()方法的比较

wait():

  • 来自Object类
  • 释放锁 该进程进入等待对列 其他线程可以使用同步控制块或者同步方法
  • wait() 后 需要进入调度对列被系统调度或者被其他线程唤醒
  • wait notify和notifyAll只能在同步控制方法或者同步控制块里面使用 (因为是Thread类下的)
  • wait notify和notifyAll不需要捕获异常

sleep():

  • 来自Thread类
  • 没有释放锁 占着CPU去睡觉 其他线程无法调用系统资源(CPU) OS认为线程正在工作
  • sleep() 后 睡觉结束以后直接会运行
  • sleep可以在任何地方使用 (因为是Object类下的)
  • sleep必须捕获异常

生产者和消费者

生产者-消费者问题(有限缓冲问题)是典型的线程通信(多线程同步)问题

该问题描述了共享固定大小缓冲区的两个线程 (即所谓的“生产者”和“消费者” )在实际运行时会发生的问题

生产者的主要作用是生成一定量的数据放到缓冲区中，然后重复此过程。与此同时，消费者也在缓冲区消耗这些数据。该问题的关键就是要保证生产者不会在缓冲区满时加入数据，消费者也不会在缓冲区中空时消耗数据

需要注意的问题:

  • 当缓冲区为空时 消费者发现缓冲区空 则等待并提醒生产者生产
  • 当缓冲区为满时 生产者发现缓冲区满 则等待并提醒消费者消费

虚假唤醒

虚假唤醒是一种现象，它只会出现在多线程环境中，指的是在多线程环境下，多个线程等待在同一个条件上，等到条件满足时，所有等待的线程都被唤醒，但由于多个线程执行的顺序不同，后面竞争到锁的线程在获得时间片时条件已经不再满足，线程应该继续睡眠但是却继续往下运行的一种现象

多线程环境的编程中，我们经常遇到让多个线程等待在一个条件上，等到这个条件成立的时候我们再去唤醒这些线程，让它们接着往下执行代码的场景。假如某一时刻条件成立，所有的线程都被唤醒了，然后去竞争锁，因为同一时刻只会有一个线程能拿到锁，其他的线程都会阻塞到锁上无法往下执行，等到成功争抢到锁的线程消费完条件，释放了锁，后面的线程继续运行，拿到锁时这个条件很可能已经不满足了，这个时候线程应该继续在这个条件上阻塞下去，而不应该继续执行，如果继续执行了，就说发生了虚假唤醒。

为了防止虚假唤醒 设置多个进程进行测试 使用while进行判断

小结

synchronized实现同步的基础: java中的每一个对象都可以作为锁

当一个线程试图访问同步代码块时，他首先必须得到锁，退出或者是抛出异常时必须释放锁

普通方法和同步锁无关 普通方法无需等待同步锁 即普通方法后发现和同步锁无关

锁的三种形式:

  • 对于普通同步方法，锁的是当前实例对象
    • synchronized关键字锁定的是调用者(this 调用该方法的对象)
  • 对于静态同步方法，锁的是当前的Class对象
    • 某单对象里面如果有多个synchronized方法 则某一时刻只有一个同步方法能被调用 其他线程都需要等待
    • static关键词修饰的同步方法锁定的是类(class 类对象)
    • 因为当前的Class对象只有一个 所以所有的静态同步方法用的也是同一把锁
  • 对于同步方法块，锁是synchronized括号里面的配置对象

静态的同步方法与非静态的同步方法之间是不会有竞争条件 因为锁定的内容不一样

什么是Java的不安全

在Java中不安全现象是指在编程中可能存在的安全漏洞和安全隐患

Java 作为一种编程语言，具有许多优势，但也存在一些潜在的安全问题

Java 多线程下的不安全是指在多线程并发执行的情况下，可能会存在一些潜在的安全漏洞和隐患

常见的 Java 多线程下的不安全现象：

  • 线程安全问题：多线程并发执行可能会导致数据不一致、死锁等问题。
    • 线程安全的数据结构（如 ConcurrentHashMap等）可以确保多线程下的数据一致性和正确性
  • 线程同步问题：多线程并发执行时，需要使用同步机制来保证数据的一致性和正确性
    • 使用 synchronized 关键字或 ReentrantLock 类来保证线程同步。但过度同步可能导致性能下降。
  • 线程间通信问题：多线程之间需要进行通信时，可能会存在数据竞争、数据不一致等问题。
    • 多线程并发访问共享数据时，需要使用适当的同步机制来保证数据的一致性和正确性。
  • 死锁问题：多线程并发执行时，可能会存在死锁现象。
    • 当多个线程分别持有其他线程需要的资源时，会发生死锁。为了避免死锁，需要确保线程之间的资源请求和释放遵循一定的顺序。
  • 线程泄漏问题：多线程并发执行时，可能会存在线程泄漏问题
    • 当一个线程无法正常终止时，可能会导致系统资源泄露。为了避免线程泄漏，需要确保线程的正确终止。

为避免多线程下的不安全问题，建议在编写 Java 代码时遵循最佳实践，例如使用线程安全的数据结构、使用同步机制来保证线程同步、确保线程之间的通信正确、避免死锁和线程泄漏等。

写入时复制思想

写入时复制(CopyOnWrite)思想 简称为COW思想 是计算机程序设计领域中的一种优化策略

若有多个调用者(Callers)同时要求相同的资源(如内存或者是磁盘上的数据存储)，他们会共 同获取相同的指针指向相同的资源，直到某个调用者视图修改资源内容时，系统才会真正复制一份专用 副本(private copy)给该调用者，而其他调用者所见到的最初的资源仍然保持不变

该过程对其他调用者都是透明的(transparent) 即不可知的

当调用者如果没有修改资源 则都指向同一个资源 若有调用者修改资源 则调用者会指向资源副本 其他调用者仍然访问的是同一个资源 这样就不会造成线程不安全现象

Vector和CopyOnWriteArray

Vector是增删改查方法都加了synchronized 保证同步 但是性能较弱

CopyOnWriteArrayList 只是在增删改上加锁 但是读不加锁 在读方面的性 能就较好 支持读多写少的并发情况
