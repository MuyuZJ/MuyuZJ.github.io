什么是DB

DB 即Database 数据库 是长期存放并管理在计算机内有组织可共享的大量数据的”仓库”

数据库分为关系型数据库和非关系型数据库

关系型数据库和非关系型数据库

关系型数据库和非关系型数据库是数据库中两种不同的数据存储和管理方式

  • 关系型数据库
    • 典型的关系型数据库: MySQL / Oracle / Microsoft SQL Server
    • 关系型数据库是基于表格的形式存储数据 表格由行和列组成 每个单元格包含一个数据项
  • 非关系型数据库
    • 典型的非关系型数据库: Redis / MongoDB
    • 非关系型数据库则不使用表格的形式存储数据 而是使用键值对(key-value pairs)的形式存储数据

记录和字段

  • 记录是一个数据集中的一个具体的数据项 通常包含一个或多个字段
  • 字段是记录中用于存储数据的组成部分 它们具有特定的数据类型和属性

例如: 在一个学生数据表中

  • 一个学生有编号,姓名,年龄等诸多数据 每一个项目就是一个字段
  • 拥有这些数据项(字段) 的学生有很多个 则有多条记录

MySQL数据管理

外键

若公共关键字在一个关系(从表)中为主键 则该关键字称为另一个关系(主表)的外键

外键存在为了保持数据的一致性和完整性 通过设置数据的约束关系 使两张表进行关联 主表的外键只能引用从表的主键列或者空值

建表的时候可以设定当前表(主表)的某字段是另一个表(从表)的外键(foreign key)

逻辑外键和物理外键

物理外键是指在表的定义中使用外键约束来定义的键,它与该键所引用的表中的某一列相关联。物理外键约束要求引用表中的数据必须与被引用表中的数据相匹配,否则插入或更新操作将会失败。物理外键约束可以保证引用完整性,但可能会影响查询性能,因为它需要进行额外的比较操作

逻辑外键是指在表的定义中不使用外键约束,但在查询时使用约束来定义的键。逻辑外键通常使用关系型数据库中的连接条件来实现,它与引用表中的某一列相关联。逻辑外键约束要求引用表中的数据必须与被引用表中的数据相匹配,否则查询结果可能不正确。逻辑外键约束不会影响查询性能,因为它不需要进行额外的比较操作。

通常普通使用会选择使用逻辑外键 需要数据一致性和完整性的情况下考虑使用物理外键

删除具有主外键关系的表时 需要先删除子表 再删除主表

事务

什么是事务

事务就是将一组SQL语句放在同个批次执行 如果同批次SQL语句有出错则所有SQL语句都被取消执行 恢复原状

MySQL事务只支持InnoDB和BDB数据库引擎

事务ACID原则

知乎 - ACID原则

ACID原则 对应的是 原子性(Atomic) 一致性(Consist) 隔离性(Isolated) 持久性(Durable)

  • 原子性(Atomic)
    • 事务内 同一批次的SQL语句要么都执行 要么都不执行
  • 一致性(Consist)
    • 事务必须始终保持系统处于一致的状态 不管在任何给定的时间并发事务有多少
    • 理解为 需要保持最终结果一致性
  • 隔离性(Isolated)
    • 串行化或序列化请求 同一个时刻只有一个请求使用同一组数据 无法查看其他执行中事务正在操作的数据
  • 持久性(Durable)
    • 事务完成后 应该被持久化保存 不会回滚 即使系统崩溃或断电都不会丢失数据

事务实现

MySQL默认开启自动提交事务

使用事务前需要设置手动提交(关闭自动提交模式) 结束(提交或回滚)后需要开启自动提交事务(恢复原状态)

常见的事务问题

  1. 脏读:脏读指一个事务读取了另外一个事务未提交的数据

索引

索引介绍

索引:

  • 提高查询速度 (数据量少的时候感知不明显 数据量大时索引就很重要了)
  • 确保数据唯一性
  • 加速表连接 实现表之间的参照完整性
  • 减少分组和排序时间
  • 全文检索字段进行搜索优化
  • 建立/重新建立索引时间较长 不建议添加在经常变动记录的表中

分类:

  • 主键索引 primary key
  • 唯一索引 unique
  • 常规索引 index
  • 全文索引 fulltext

可以使用explain关键字来查看语句执行性能

参考 CSDN - 看懂explain

主键索引

作用: 记录的某一个属性(字段)可以唯一标识该记录 通常作为主键索引

特点:

  • 最常见 (就是主键)
  • 确保数据的唯一性
  • 确定特定记录在数据库之中的位置

唯一索引

作用: 避免同一个表中的数据列中的值重复

特点:

  • 唯一索引可以有多个(某字段中的值不重复 可以设置多个字段) (主键: 只有一个主键 只能设置一个)

常规索引

作用: 快速定位特定数据

特点:

  • index和key关键字 都可以设置常规索引 添加在查询经常找条件的字段

全文索引

作用: 快速定位特定数据

数据库规范化设计

什么是数据库规范化设计

数据库规范化设计（Normalization）是指在关系型数据库中，通过减少数据冗余、提高数据一致性和完整性，使数据库中的数据保持合理的数据结构，便于管理和维护的过程。数据库规范化的主要目的是为了确保数据完整性和一致性，避免数据不一致、数据冗余等问题

数据库规范化设计是关系型数据库设计的重要环节

在关系型数据库中，数据由表和列组成。为了实现数据的有效性和一致性，需要遵循一定的规则。

  1. 保证每个表只包含一个实例，每个实例只包含一个属性
  2. 每个表中的列都是唯一的
  3. 每个表中的数据都是关于该表的主题的，并且与该表的其他列保持一致
  4. 每个表中的数据都是关于该表的主题的，并且与该表的其他表保持一致

数据库规范化设计包括以下几个步骤：

  1. 确定实体和属性：将数据库中的数据分为实体和属性，实体表示数据的主要对象，属性表示实体具有的特征
  2. 确定主键：确定每个实体或关系的主键，主键是唯一的标识一个实例的属性。
  3. 确定关系：确定实体之间的关系，将实体之间的关系定义为关系。
  4. 确定约束：确定每个列的约束条件，如非空、唯一等。
  5. 确定表结构：根据实体和关系，确定每个表的结构，将实体和关系映射到表中。

为何要规范化设计

当数据库比较复杂的时候就需要进行数据库规范化设计

良好的数据库设计 糟糕的数据库设计
数据冗余 较少 较多
存储空间 相对少 合理 较多 不合理
数据更新和插入异常 无 可能有
程序性能 较好 方便开发 差

良好的数据库设计不一定要求没有冗余

不合理的数据库设计会导致: 信息重复 更新/插入/删除异常

数据库规范化设计的步骤

数据库规范化设计包括以下几个步骤：

  1. 确定实体和属性：将数据库中的数据分为实体和属性，实体表示数据的主要对象，属性表示实体具有的特征
  2. 确定主键：确定每个实体或关系的主键，主键是唯一的标识一个实例的属性。
  3. 确定关系：确定实体之间的关系，将实体之间的关系定义为关系。
  4. 确定约束：确定每个列的约束条件，如非空、唯一等。
  5. 确定表结构：根据实体和关系，确定每个表的结构，将实体和关系映射到表中。

三大范式

参考 CSDN - 数据库设计之规范化理论

不同等级的范式(Normal Form, NF)做不同的事情

范式分为: 一范式 二范式 三范式 BC范式 四范式 五范式 (一级比一级有更严格的要求)

通常数据库会规范化到三范式或BC范式

各范式存在包含关系

满足高级范式必定满足低级范式

一范式

第一范式是最基本的规范形式，即关系中每个属性值都满足原子性

但是满足第一范式的关系模式并不一定是一个好的关系模式

如果关系模式R,其所有的属性均为简单属性，即每个属性都城是不可再分的，则称R属于第一范式,简称1NF

适用于数据量相对简单的场景，如网站注册、新闻分类等

数据表的原子性:

  • 表中同一列的类型相同
  • 一个列名只对应一列
  • 每一列不可再分
  • 行(记录) 上下不受影响

二范式

如果关系模式R∈ 1NF,且每个非主属性都完全函数依赖于R的每个关系键,则称R属于第二范式

适用于需要考虑联合主键的场景，如订单商品、学生成绩等

三范式

它是R∈ 2NF 且不包含传递依赖项（非键属性通过另一个非键属性依赖于主键）,则称R属于第三范式

消除了数据冗余/更新异常/插入异常/删除异常等诸多问题

对于大多数表，第三范式通常就足够了，因为它避免了最常见的一种数据异常

适用于需要避免数据冗余和数据不一致的场景，如教师授课、图书分类等

传递依赖:

若 A -> B 且 B -> C 则称为C传递依赖于A

BC范式

BC范式(Boyce-Codd Normal Form, 巴斯-科德范式) 是对三范式的优化

三范式可能存在主属性对候选键的部分依赖和传递依赖，因此关系模式的分离仍不够彻底

BC范式在函数依赖的范畴内，已经实现了模式的彻底分解，消除了产生插入异常和删除异常的根源，而且数据冗余也减少到极小程度

适用于需要保证数据一致性的场景，如账户管理、库存管理等

四范式

四范式是Boyce-Codd的标准格式且不包含一个以上的多值依赖项

奇异化指的是为了消除多值依赖而剥离出来的列，如果发现这些列之间存在依赖关系（称为“连接依赖”），则需要再次拆分为独立的表进行处理

适用于需要解决多值依赖的场景，如油价查询、课程表查询等

五范式

在4NF的基础上，消除奇异化（即超规范化）

适用于需要消除奇异化的场景，如多对多关联关系、产品分类等

数据库规范化

什么是规范化

各范式级别是在分析函数依赖条件下对关系模式分离程度的一种测度，范式级别可以逐级升高

一个低一级范式的关系模式，通过模式分解转化为若干个高一级范式的关系模式的集合,这种分解过程叫作关系模式的规范化(Normalization)

实际场景中需要综合考虑使用哪些范式，并根据具体情况进行适度的退化或冗余设计

规范化目的和原则

规范化的目的就是使结构合理，消除存储异常，使数据冗余尽量小，便于插入、删除和更新

规范化的基本原则就是遵从概念单一化“一事一地”的原则，即一个关系只描述一个实体或者实体间的联系

规范化步骤

规范化就是对原关系进行投影，消除决定属性不是候选键的任何函数依赖。具体可以分为以下几步:

  1. 对1NF关系进行投影，消除原关系中非主属性对键的部分函数依赖，将1NF关系转换成若干个2NF关系。
  2. 对2NF关系进行投影，消除原关系中非主属性对键的传递函数依赖，将2NF关系转换成若干个3NF关系。
  3. 对3NF关系进行投影，消除原关系中主属性对键的部分函数依赖和传递函数依赖，也就是说使决定因素都包含一个候选键。得到一组BCNF关系

一般来说 规范化到BCNF即可 再往后规范其实没很大必要

数据库池化技术

数据库池化技术是一种在应用程序中重用数据库连接的技术。数据库池化可以提高应用程序的性能和减少数据库连接的开销。当应用程序需要连接到数据库时，它可以从池中获取一个已存在的连接，而不是创建一个新的连接。当连接使用完毕后，它将被返回到池中，以便下一次使用。

简而言之就是预先设定一个存储连接的池子 保持连接 需要使用的时候从池子获取连接 用完归还不关闭 这样就避免了频繁地创建和关闭 只需要使用与放回即可

JNDI 可以在应用程序中查找 Web 服务、数据库连接或消息队列等

  • c3p0
  • c3p0-sources
  • mchange-common-java
